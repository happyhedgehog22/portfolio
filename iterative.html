Iterative Design

Background

This project was done for CS1300: UI/UX. The goal of the assignment was to design an interface for an existing startup without looking at the startup’s current UI. We went through several rounds of design and feedback to ensure we had the best design.

The startup we chose was Fetii. A company that uses large passenger vans and buses to transport large groups of people. Similarly to Uber, users can book rides for now or later depending on their needs, but instead of one person paying with Fetii it is much easier to split the bill. Users can scan a QR code before getting on a vehicle to split the ride fare. Fetii targets a large consumer group ranging from corporate groups attending conferences to greek life groups. In this project, we designed a new desktop website for Fetii where users can book rides, find out about the company, and apply to work as a driver for Fetii.

Wireframing

Based on our sketches, we made wireframes based on the following design decisions

  • Feature both Rider and Driver functionality. Need a clear and distinct navbar that shows the user where they currently are on the page.
  • Ordering a van should have a minimalist design and highlight the key items: pickup/dropoff location and time and a map for the user to visualize the route
  • Pickup Now and Pickup Later should look very similar, the user should understand what to do either way

Step 2: Hi-Fi Designs

Initial Mockup

We then transformed our wireframes into an interactive high-fidelity mockup. Click around to the different pages!

Critique Feedback

We had an in-class critique of our initial high-fi mockup. We received a lot of useful feedback that we used to improve our design.

  • "Request a Ride” page was unclear: In general, users wanted a clearer and more intuitive experience, as well as more information about the ride. Specifically, users found it unclear how to set a location, pay for the ride, specify the number of riders, and did not find an estimated price.
    • To fix this we added a place for users to enter the number of passengers and estimated cost on the initial ride request page
  • No way to view account information: Users wanted a Profile page to keep track of their scheduled and past rides, as well as their payment information.
    • To fix this we added a profile page that showed the user information about their account and ride history
  • More information: Users said they would have appreciated more information about the company and the ride both before and after it had been requested
    • To fix this we added more ride details through the ride requesting process

Step 3: Usertesting

Usertesting is a platform where users can test out prototypes and designers can get feedback in the earlier stages to make adjustments to their products

Tasks

These were the tasks that our testers were given.

Scenario: Imagine that you need to transport yourself and a group of others. Book a ride for the group. This platform is a prototype so some elements may be prefilled and not able to be interacted with.

  • Without leaving the homepage, what are your initial impressions of the website?
  • Book a ride for now
  • Submit a job application on this website
  • Set up an account on this website

After each task, the user was asked two questions

  1. Did you complete the task successfully?
  2. Rate this task overall from very difficult to very easy

Usertesting Feedback

Here is a summary of the feedback we received from the three users who tested our prototype

Initial impressions

  • Most prominent features: the color purple, the name of the company, the request a ride button, the sign up to drive button, and the nav bar
  • Appears to have everything they would expect from a rideshare company
  • One user mentioned items in the nav bar are not level
  • One user mentioned that they thought the my account button should be an icon

Book a ride for now

  • 2 out of 3 users were able to complete the task with little to no difficulty
  • One user never hit confirm
  • One user mentioned they would like to see the estimated pick up time before the confirmation screen
  • All noticed you could navigate via the nav bar or the home screen

Submit an application

  • All users were able to easily complete the task
  • One user mentioned the photo on the confirmation page was not inclusive

Set up an account

  • All three users were able to complete the task easily
  • One user questioned whether past drives would expand to scroll on the page or link to another page

Future Changes

Based on the feedback received in Usertesting these are changes we will implement in the future

  • Make the nav bar smaller and have all nav bar contents level within
  • Show the estimated pick up time before the user needs to confirm the ride
  • Change photo on confirmation of application screen to be more diverse and inclusive
  • Create a past drives page

Conclusion

This project taught me the importance of reworking design and seeking outside feedback. Even after improving our design based on feedback from the critique we still found a number of issues from our Usertesting research.
